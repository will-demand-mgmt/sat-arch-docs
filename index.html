External clients

Customer OSS (Operations Support Systems) and partner systems that consume this API. They submit satellite capacity requests over REST, authenticated via OAuth2 Bearer tokens.

Typical payload: beam ID, frequency band, polarisation, requested time window, quantity of capacity, and a contract reference.

RESTOAuth2OSSpartner systems

API Management

The public entry point. Validates OAuth2 tokens, enforces per-client rate limits and throttling policies, and rejects malformed requests before they reach the domain layer.

Acts as a shield — the orchestration engine only sees well-formed, authenticated requests.

AuthN/AuthZrate limitingschema validation

Orchestration engine (Saga)

Manages the full request lifecycle as a distributed saga. Coordinates the sequence: validate → decide → commit. Each step is idempotent, with explicit retry, timeout, and compensation logic in case a downstream call fails mid-flow.

If validation succeeds but the commit to CAPI fails, the saga triggers a rollback of any partial reservations already made.

saga patternstate machinecompensationidempotency

Validation engine

Runs three independent checks: contract validity (via GCRM), beam eligibility (via OSS), and capacity availability (via CAPI). All three must pass for the request to proceed to the decision stage.

GCRMOSSCAPI

Decision engine

Compares requested capacity against validated availability and produces one of three outcomes: APPROVED (full capacity available), PARTIAL (partial capacity available, alternative proposal generated), or REJECTED (no capacity, reason attached).

In PARTIAL/REJECTED cases, the engine may suggest an alternative beam, time window, or reduced allocation.

APPROVEDPARTIALREJECTED

State & Data management

Persists all request state transitions in a database acting as a state machine. States: PENDING → VALIDATING → DECIDING → COMMITTED.

Maintains a contract cache (to avoid repeated GCRM calls) and a capacity snapshot cache (to reduce CAPI load). All transitions are written to an immutable audit log.

state machinecacheaudit log

Event backbone

Pub/Sub event bus for async communication between internal components and external consumers. Every significant state change produces an event.

Key events: request.received, request.approved, request.rejected, capacity.reserved.

pub/subasyncevent-driven

GCRM adapter — Contracts

Reads active customer contracts from the CRM system. Validates that the requested beam is within the contracted scope (region, service tier, authorised beams list).

CAPI adapter — Capacity inventory

Queries the capacity inventory system for a real-time snapshot of available bandwidth on a given beam. Returns current free capacity before bookings are applied.

CP adapter — Capacity planner

Queries the capacity planning system to account for existing bookings in the requested time window. Subtracts committed capacity from the CAPI snapshot to produce net available capacity.

OSS / Frequency Planning adapter

Interfaces with the Operations Support System for two checks: geographic footprint verification (is the ground station inside the beam coverage area?) and technical compatibility (frequency, polarisation, service type).

The independent legacy and third-party systems of record. The integration layer above them keeps all coupling contained in adapter code — the domain layer has no direct knowledge of their APIs or data formats.

Incoming request payload

The raw request from an external client, containing: target beam ID, frequency band and polarisation, requested time window (start/end), capacity quantity (MHz or Mbps), and contract reference ID.

beam IDfrequencytime windowcontract ref

Geographic check

Verifies that the customer's ground station coordinates fall within the satellite beam's footprint — the geographic area on Earth that the beam can serve. This is a geometric calculation against coverage maps maintained in OSS/Frequency Planning.

OSSfootprint polygonground station

Contract check

Reads the customer's active contract from GCRM and verifies that the requested beam is explicitly authorised. Contracts may restrict access by region, beam tier, or service type.

GCRMauthorised beamsservice tier

Technical check

Confirms that the requested frequency, polarisation (horizontal/vertical/circular), and service type are physically compatible with the target beam's specification. A mismatch here means the signal cannot be transmitted or received correctly.

OSSfrequency planpolarisation

Eligibility gate

All three checks (geographic, contractual, technical) must pass. A single failure marks the beam as ineligible. The decision engine then searches for an alternative eligible beam before issuing a REJECTED response.

all pass → proceedany fail → alt. beam

Alternative beam proposal

When the requested beam is not eligible, the decision engine searches for nearby beams that satisfy all three checks and have sufficient capacity. If found, a PARTIAL response is returned with the alternative proposal. If not, REJECTED.

PARTIALREJECTED

CAPI — instantaneous snapshot

Provides a real-time view of total bandwidth currently available on the beam, before any time-windowed bookings are applied. This is the ceiling from which committed capacity is subtracted.

real-timebandwidth ceiling

CP — bookings in time window

Queries all confirmed reservations that overlap with the requested time window on the target beam. Subtracts committed capacity from the CAPI snapshot to yield net available capacity for this request.

time-awarecommitted capacity

MCM — operational constraints

Applies mission-level restrictions: scheduled maintenance windows, orbital manoeuvre blackouts, and government/military priority reservations that preempt commercial availability regardless of CAPI/CP readings.

mission opsblackout windowspriority

Capacity gate

Compares the net available capacity (CAPI − CP bookings − MCM constraints) against the requested quantity. Full match → APPROVED. Partial match → PARTIAL with a reduced proposal. No match → REJECTED.

full → APPROVEDpartial → PARTIALnone → REJECTED

APPROVED

Full requested capacity is available. The saga proceeds to commit: CAPI is updated with the reservation, the state machine transitions to COMMITTED, and a capacity.reserved event is published.

commit to CAPIcapacity.reserved event

PARTIAL

Only a portion of the requested capacity is available. The decision engine generates an alternative proposal (reduced quantity, different time slot, or alternative beam) and returns it alongside the PARTIAL status for the client to accept or decline.

alt. proposalclient decision required

REJECTED

No capacity available on the requested beam (or any alternative) within the requested window. The response includes a structured reason code and, where possible, a suggested alternative time window.

reason codealt. time suggestion

Event backbone — output

Every outcome publishes a structured event to the bus. Downstream consumers (billing, monitoring, customer notifications) subscribe independently without coupling to this service.
