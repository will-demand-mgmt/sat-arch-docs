Satellite Capacity
Orchestration Platform

A unified orchestration layer that coordinates capacity requests across the full satellite ground segment — from customer order to beam reservation — in a single, auditable workflow.

What this system does
📡

Beam eligibility validation

Verifies that a customer's ground station is within the beam footprint, that the contract authorises the beam, and that frequency and polarisation are technically compatible.

📊

Real-time capacity check

Combines CAPI inventory snapshots, CP booking commitments, and MCM mission constraints to compute net available capacity for any beam and time window.

⚙️

Distributed saga orchestration

Coordinates multi-step workflows with retry, timeout, and compensation logic. If any step fails mid-flow, the saga rolls back partial reservations automatically.

External clients

Customer OSS (Operations Support Systems) and partner systems that consume this API. They submit satellite capacity requests over REST, authenticated via OAuth2 Bearer tokens.

Typical payload: beam ID, frequency band, polarisation, requested time window, quantity of capacity, and a contract reference.

RESTOAuth2OSSpartner systems

API Management

The public entry point. Validates OAuth2 tokens, enforces per-client rate limits and throttling policies, and rejects malformed requests before they reach the domain layer.

Acts as a shield — the orchestration engine only sees well-formed, authenticated requests.

AuthN/AuthZrate limitingschema validation

Orchestration engine (Saga)

Manages the full request lifecycle as a distributed saga. Coordinates the sequence: validate → decide → commit. Each step is idempotent, with explicit retry, timeout, and compensation logic in case a downstream call fails mid-flow.

If validation succeeds but the commit to CAPI fails, the saga triggers a rollback of any partial reservations already made.

saga patternstate machinecompensationidempotency

Validation engine

Runs three independent checks: contract validity (via GCRM), beam eligibility (via OSS), and capacity availability (via CAPI). All three must pass for the request to proceed to the decision stage.

GCRMOSSCAPI

Decision engine

Compares requested capacity against validated availability and produces one of three outcomes: APPROVED (full capacity available), PARTIAL (partial capacity available, alternative proposal generated), or REJECTED (no capacity, reason attached).

In PARTIAL/REJECTED cases, the engine may suggest an alternative beam, time window, or reduced allocation.

APPROVEDPARTIALREJECTED

State & Data management

Persists all request state transitions in a database acting as a state machine. States: PENDING → VALIDATING → DECIDING → COMMITTED.

Maintains a contract cache (to avoid repeated GCRM calls) and a capacity snapshot cache (to reduce CAPI load). All transitions are written to an immutable audit log.

state machinecacheaudit log

Event backbone

Pub/Sub event bus for async communication between internal components and external consumers. Every significant state change produces an event.

Key events: request.received, request.approved, request.rejected, capacity.reserved.

pub/subasyncevent-driven

Incoming request payload

The raw request from an external client, containing: target beam ID, frequency band and polarisation, requested time window (start/end), capacity quantity (MHz or Mbps), and contract reference ID.

Geographic check

Verifies that the customer's ground station coordinates fall within the satellite beam's footprint — the geographic area on Earth that the beam can serve. This is a geometric calculation against coverage maps maintained in OSS/Frequency Planning.

OSSfootprint polygonground station

Contract check

Reads the customer's active contract from GCRM and verifies that the requested beam is explicitly authorised. Contracts may restrict access by region, beam tier, or service type.

GCRMauthorised beamsservice tier

Technical check

Confirms that the requested frequency, polarisation (horizontal/vertical/circular), and service type are physically compatible with the target beam's specification. A mismatch here means the signal cannot be transmitted or received correctly.

OSSfrequency planpolarisation

Eligibility gate

All three checks (geographic, contractual, technical) must pass. A single failure marks the beam as ineligible. The decision engine then searches for an alternative eligible beam before issuing a REJECTED response.

all pass → proceedany fail → alt. beam

Alternative beam proposal

When the requested beam is not eligible, the decision engine searches for nearby beams that satisfy all three checks and have sufficient capacity. If found, a PARTIAL response is returned with the alternative proposal. If not, REJECTED.

PARTIALREJECTED

CAPI — instantaneous snapshot

Provides a real-time view of total bandwidth currently available on the beam, before any time-windowed bookings are applied. This is the ceiling from which committed capacity is subtracted.

real-timebandwidth ceiling

CP — bookings in time window

Queries all confirmed reservations that overlap with the requested time window on the target beam. Subtracts committed capacity from the CAPI snapshot to yield net available capacity for this request.

time-awarecommitted capacity

MCM — operational constraints

Applies mission-level restrictions: scheduled maintenance windows, orbital manoeuvre blackouts, and government/military priority reservations that preempt commercial availability regardless of CAPI/CP readings.

mission opsblackout windowspriority

Capacity gate

Compares the net available capacity (CAPI − CP bookings − MCM constraints) against the requested quantity. Full match → APPROVED. Partial match → PARTIAL with a reduced proposal. No match → REJECTED.

full → APPROVEDpartial → PARTIALnone → REJECTED

APPROVED

Full requested capacity is available. The saga proceeds to commit: CAPI is updated with the reservation, the state machine transitions to COMMITTED, and a capacity.reserved event is published.

commit to CAPIcapacity.reserved event

PARTIAL

Only a portion of the requested capacity is available. The decision engine generates an alternative proposal (reduced quantity, different time slot, or alternative beam) and returns it alongside the PARTIAL status for the client to accept or decline.

alt. proposalclient decision required

REJECTED

No capacity available on the requested beam (or any alternative) within the requested window. The response includes a structured reason code and, where possible, a suggested alternative time window.

reason codealt. time suggestion

Every outcome publishes a structured event to the bus. Downstream consumers (billing, monitoring, customer notifications) subscribe independently without coupling to this service.

Happy Path Full Approval Flow
click to expand ↗
Request phase
The client submits a capacity request. The Validation Engine runs all three checks (contract, beam eligibility, capacity) in parallel against GCRM and CAPI. All pass → Decision = APPROVED.
Acceptance phase
The client explicitly accepts the approved proposal via a separate API call, triggering the execution phase.
Execution phase
The Orchestrator runs three sequential steps against CP, OSS, and MCM. On full success, Status = COMPLETED and a capacity.reserved event is published to the event backbone.
Partial Flow Client Interaction Loop
click to expand ↗
Partial decision
CAPI returns less capacity than requested. The Orchestrator computes an alternative proposal (reduced quantity, different time slot, or alternative beam) and returns it to the client alongside the PARTIAL status.
Client accepts
The client explicitly accepts the alternative via POST /accept. The Orchestrator continues the execution workflow as in the Happy Path.
Client rejects
No action or explicit cancel. The Orchestrator marks the request as CANCELLED. A request.partial event is published regardless of the client's choice.
Failure Path Execution with Rollback
click to expand ↗
Execution steps
The saga executes three sequential steps: CP (frequency plan update), OSS (transmission plan generation), MCM (config apply). Steps 1 and 2 succeed; Step 3 fails.
Saga rollback
On Step 3 failure, the Orchestrator triggers compensating transactions in reverse order: revert OSS first, then revert CP. Each compensation restores the downstream system to its pre-execution state.
Final state
Status = ROLLED_BACK. A request.failed event is published to the backbone. The client receives a failure notification with the ROLLED_BACK status.
Validation Failure Early Rejection
click to expand ↗
Early exit
Validation failure short-circuits the flow before execution. No capacity is reserved, no downstream systems are touched beyond the validation read.
Failure reasons
Three possible rejection codes: CONTRACT_EXCEEDED (contract does not authorise the beam or service), BEAM_NOT_ELIGIBLE (geographic, technical, or contractual mismatch), NO_CAPACITY (insufficient bandwidth after bookings and MCM constraints).
Event
A request.rejected event with the reason code is published to the backbone. The client receives the rejection with a structured reason code.
Edge Cases — failure modes, race conditions & operational scenarios
Edge Case Concurrency Control
click to expand ↗
Soft reservation with TTL
When request A is validated and approved, CAPI applies a soft reservation with a TTL (e.g. 5 min). This is not a committed reservation — it becomes hard only after the client's explicit accept and execution commit.
Concurrent visibility
Request B sees reduced availability (60 MHz) because A's soft reservation is immediately visible in CAPI. This prevents double-booking without requiring distributed locks.
TTL expiry
If client A does not accept within the TTL window, the soft reservation is released automatically — see the Expired Request edge case. Capacity returns to the pool and becomes visible to future requests.
Edge Case Accept Delay & Revalidation
click to expand ↗
Soft reservation during delay
After APPROVED, a soft reservation is held in CAPI until the TTL expires. If the client accepts within the TTL, the reservation is still active. If TTL expires before accept, the request transitions to EXPIRED — see the Expired Request edge case.
Revalidation on accept
The Orchestrator always re-checks capacity before triggering execution. If capacity has changed (other requests committed in the interim), the result may differ from the original approval.
Invalidation outcomes
If revalidation fails: the request is marked INVALIDATED and the client receives either a rejection with reason or a new PARTIAL proposal if reduced capacity is still available.
Edge Case Timeout & Retry with SLA Budget
click to expand ↗
SLA budget
A global SLA timer (e.g. 15 min) is started at execution begin. This budget is shared across all retry attempts — not reset per retry. If the budget is exhausted before success, rollback is triggered regardless of remaining retries.
Exponential backoff
Retries use exponential backoff: 2s → 4s → 8s between attempts (max 3). This avoids overwhelming a struggling downstream system with rapid successive calls.
Failure outcome
If all retries are exhausted, the saga triggers a rollback of any steps already completed. The client receives a FAILED notification. The event request.failed is published to the backbone.
Edge Case Rollback Failure — Critical State
click to expand ↗
Rollback failure
MCM fails both on execution and on compensation. OSS and CP have been successfully reverted, but MCM remains in an inconsistent state — its config was never applied but the revert also failed.
Critical alert
Status = ROLLBACK_FAILED. A system.critical_alert event is published to the backbone. This triggers immediate notification to on-call operators — the system cannot self-recover from this state.
Manual reconciliation
The CP and OSS teams must verify their systems are consistent. The MCM team must manually inspect and revert or confirm the config state. A post-incident review should produce a runbook for this failure mode.
Edge Case Idempotency Handling
click to expand ↗
Idempotency key
Each request carries a client-generated idempotency key (e.g. UUID). The API layer checks the key against the Idempotency Store before forwarding to the Orchestrator.
Duplicate handling
If the key is found and not expired, the cached response is returned immediately — the request is never forwarded to the Orchestrator. This is safe for retries caused by network timeouts or client-side failures.
Key TTL: 24h
The key expires after 24 hours. After expiry, the same key is treated as a brand-new request. This prevents the store from growing unbounded while covering all realistic retry windows.
Edge Case Expired Request Handling
click to expand ↗
Scheduler trigger
A background scheduler runs periodically (e.g. every 60s) and scans the request DB for APPROVED requests older than the TTL threshold. This is not event-triggered — it is a polling job.
Reservation release
For each expired request, the soft reservation is released on CAPI. Capacity returns to the pool immediately and becomes visible to new incoming requests — preventing capacity from being locked indefinitely.
Event published
A request.expired event is published to the backbone. Downstream subscribers (e.g. notification service, analytics) can react accordingly. The client is not actively notified unless they poll the request status endpoint.

Five non-negotiable principles to enforce across the entire development team.

1
Idempotency everywhere
Every step must be safe to retry. If a step is called twice with the same input, the result must be identical and side effects must not be doubled. This applies to API calls, saga steps, and compensation actions.
→ Idempotency store · request_hash · cached response
2
Stateless services + centralised state
No state lives inside microservices. All request state, execution steps, reservations and audit events are persisted in the central DB. Any service instance can pick up any request at any point without coordination.
→ CapacityRequest · ExecutionStep · optimistic locking (version field)
3
Isolated adapters
No business logic in connectors. Adapters are responsible only for protocol translation, retry, circuit breaking, and error mapping. Business decisions belong exclusively to the Orchestration layer.
→ GCRM · CAPI · CP · OSS · MCM adapters
4
Restartable workflows
The saga can resume from any intermediate step without re-executing completed steps. Each ExecutionStep records its own status independently. A crashed orchestrator can restart and continue exactly where it left off.
→ ExecutionStep.status · next_retry_at · correlation_id
5
Observability by design
Every state transition emits a structured event on the bus. Every execution step records duration, outcome, and retry count. Every adapter call is traced with correlation_id. Without this, debugging distributed failures in production is effectively impossible.
→ AuditEvent · correlation_id · structured events · step metrics

CONTEXT
Design B is a proposal from the CDIO Architecture team (FCM 3.4 - Automated Configuration Workflow). It delegates orchestration to the existing Capacity Planner (CP) system, leverages ESM/ServiceNow for ticketing and audit, and uses MuleSoft as integration layer. It is documented here for comparative analysis and educational purposes — not as an endorsed alternative.
Architecture Overview
Gap Analysis — BRD/FRD Coverage
CRITICAL GAP No rollback / compensation mechanism — FR-12, NFR-05
The Design B diagram shows no automatic rollback sequence if Plan Configuration or OSS Resync fails. FR-12 requires that on any execution failure, the system automatically initiates a rollback to the last known stable configuration across all affected downstream systems.
RISK SCENARIO
MCM fails after frequency plans have already been generated by CP. OSS and CP are now in an updated state but MCM is not. The system is inconsistent. Without automatic rollback, an engineer must manually identify the failure, reverse the CP frequency plans, and restore MCM — potentially taking hours and requiring coordination across multiple teams. During this time the customer's capacity configuration is in an unknown state.
QUESTION FOR ARCHITECT → "How does Design B handle the case where MCM fails after frequency plans are already committed to CP? What is the recovery procedure and who owns it?"
CRITICAL GAP 15-minute SLA not enforced — NFR-01, BR-04
The CP Token Manager introduces a queue when the token is busy. There is no visible SLA timer or timeout mechanism in the design. NFR-01 requires end-to-end execution ≤15 minutes average, ≤20 minutes P95.
RISK SCENARIO
The token is held by a long-running configuration for 14 minutes. A new customer request arrives and is queued. Even if the subsequent processing takes only 3 minutes, the total elapsed time is 17 minutes — SLA breached before processing even starts. With no timer enforcing the 15-minute budget, there is no mechanism to detect, alert, or escalate SLA breaches in real time.
QUESTION FOR ARCHITECT → "How does Design B guarantee the 15-minute end-to-end SLA when requests may be queued waiting for the CP token? What happens when the SLA budget is exhausted?"
HIGH GAP No idempotency store — FR-08 retry risk
Design B has no visible idempotency mechanism. If a customer retries a request due to a network timeout, the system may process the same request twice, creating duplicate frequency plans and configuration changes on MCM.
RISK SCENARIO
Customer submits a capacity reallocation request. The network times out before receiving a response. The customer retries. Both requests reach CP and are processed. Two conflicting frequency plans are generated and applied to MCM — resulting in an invalid satellite configuration that requires manual intervention to correct.
QUESTION FOR ARCHITECT → "How does Design B prevent duplicate processing if the same request is submitted twice due to a network retry? Is there a deduplication mechanism?"
HIGH GAP DSR Engineer bypasses API Gateway — BR-07 audit gap
In the "Direct DSR Operation" flow, the DSR Engineer sends requests directly to CP without passing through MuleSoft or ESM. This creates an untracked, unauthenticated channel that violates BR-07 (full audit trail of all requests and outcomes) and NFR-08 (100% state transitions logged).
RISK SCENARIO
A DSR Engineer applies a manual configuration change directly to CP. The change is not logged in ESM, not visible in the audit trail, and not associated with any customer request. Three weeks later a customer disputes their capacity allocation. There is no record of the manual change to investigate. Compliance and contractual exposure cannot be resolved.
QUESTION FOR ARCHITECT → "BR-07 requires a full audit trail of ALL requests and outcomes including internal operations. How does the direct DSR→CP channel satisfy this requirement?"
HIGH GAP ESM/ServiceNow is not a transaction manager — NFR-05
ServiceNow is an ITSM platform — it manages tickets and service requests but does not guarantee atomicity between downstream systems. If a ticket is marked "Completed" in ESM but MCM has not confirmed the configuration, the state is inconsistent. NFR-05 requires zero partial states left without rollback.
RISK SCENARIO
CP updates the ESM ticket to "Completed" and sends the configuration to MCM. MCM accepts the request but fails silently during application. ESM shows the request as complete. The customer is notified of success. In reality the satellite configuration was never applied. The discrepancy may only be discovered during the next operational check — potentially hours or days later.
QUESTION FOR ARCHITECT → "NFR-05 requires atomic execution with zero partial states. How does Design B guarantee that ESM ticket status accurately reflects the actual state of MCM configuration at all times?"
OPEN QUESTION CP Token Manager — existing or to build?
The CP Token Manager is a key component for concurrency control. It is not clear from the design whether this is an existing capability of the Capacity Planner or a new component to be developed. This has direct impact on effort estimation and timeline.
QUESTION FOR ARCHITECT → "Is the CP Token Manager an existing component of the Capacity Planner, or is it a new component to be designed and built? If new, who builds it and what is the estimated effort?"
